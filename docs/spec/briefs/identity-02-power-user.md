# Identity-02 — Power-User / Nostr-Native

> Owner: Designer (Aditi). One of five parallel proposals. Audience: people who already know what npub, nsec, NIP-05, NIP-46 and a relay are — and want **expert affordances surfaced**, not abstracted.
> Inherits from: `App/Sources/Services/UserIdentityStore.swift`, `App/Sources/Services/Nip46/*`, `App/Sources/Services/NostrRelayService.swift`, `App/Sources/Features/Feedback/UserIdentityView.swift`, `App/Sources/Features/Feedback/Nip46ConnectCard.swift`. Coordinates with `ux-12-nostr-communication.md`, `ux-15-liquid-glass-system.md`.
> **Not** the agent identity — that surface lives at `App/Sources/Features/Settings/Agent/AgentIdentityView.swift` and is out of scope. Cross-link, do not redesign.

## 1. Stance

Five rules this brief will defend in every section:

1. **Show the kind number.** Power users read `kind: 0`, `kind: 1`, `kind: 24133` faster than friendly labels. Friendly labels go in tooltips, never in place of the truth.
2. **Show the hex.** `npub1…` is the share format; the 32-byte x-only hex is the *identity*. Both are surfaced, both are copy-targets, neither hides the other.
3. **Relays are first-class.** A signer with no relay is a tree falling in a forest. Every signed event has a `relays-acked: n/m` receipt or it didn't happen.
4. **Signer mode is a permanent badge, not a setting.** Local Key vs. Bunker is not symmetric — they have different latency, different threat models, different failure modes. The badge is everywhere the npub is.
5. **No hand-holding text.** No "your key never leaves this device" reassurance — that line belongs in identity-01-consumer. Footnotes here are factual: keychain access group, NIP number, relay name.

Voice register: terse, precise, exhaustive. Tufte-grade legibility — data-ink ratio first.

## 2. Settings → Identity Entry

Lives in the existing `SettingsView` list, *above* the agent identity row (the human signs first; the agent acts on the human's behalf).

```
┌──────────────────────────────────────────────────────────────┐
│  IDENTITY                                                     │
│  ──────────────────────────────────────────────────────────── │
│  ◐  bright-signal-a3f2                                  ⌄    │
│      npub1pl7…q9k4 · Local Key · 2 relays           ›        │
└──────────────────────────────────────────────────────────────┘
```

**Anatomy.**
- 24pt dicebear avatar (T0, 1pt hairline ring; ring goes `accent.live` red if signer is in a `failed` state, `warning` amber if a relay's last-acked age exceeds 24h, otherwise hairline).
- Generated profile name (or kind-0 `display_name` if user-edited).
- Caption row: `npubShort · {Local Key | Bunker via Amber} · {n} relays` — three semicolon-separated facts, mono font for the npub fragment.
- The trailing `⌄` chevron is a **multi-account stub** (see §4.6). MVP is single-account; the chevron remains, disabled, marked `[v2]` in code.
- Tier: T0. The row is reading content, not chrome.

**Signer mode badge**, sized `caption.small`, mono — this badge format repeats everywhere identity is shown:
- `Local Key` — neutral, no glass.
- `Bunker via Amber` (or whichever bunker pubkey resolved to a known kind-0 name) — `glass.agent` tint at T1, since remote signing is *agent-mediated infrastructure* even though the *identity* is the user. This is the only place `accent.agent` appears outside an agent surface; defended because the latency, the failure mode, and the threat model all *come from* the bunker's machine.

If the bunker pubkey has no kind-0 name we can resolve, the badge degrades to `Bunker · 7c3e…a1f2` (mono fingerprint). Never `Remote Signer` — too vague.

**Specifics.**
- **Public key block.** Three rows because power users want all three. `npub` is for sharing, `hex` is for tooling/scripting, `fingerprint` is the SHA-256 of the 32-byte hex truncated to first 16 hex chars — short enough to verify by eye in a thread, long enough to resist trivial collisions. Pick one, defend it: SHA-256 over first-16 hex because it's deterministic, doesn't reveal additional key material, and exactly fits one `mono.timestamp` row at iPhone widths.
- **Show QR** opens a sheet inheriting the `npub QR reveal` choreography from `ux-12-nostr-communication.md` §5 (320ms cascade). Single source of truth — do not redesign.
- **Profile JSON toggle.** `[JSON]` button flips the kind-0 panel from key/value rows to a syntax-highlighted JSON view in T0 / `bg.sunken` / `mono.code`, scrollable, copy-button on the trailing edge. Both views are read-only at this level; `[Edit]` opens §4.
- **Signer panel — `signs` row.** Lists kinds the active signer will produce *for this user*. Today: `kind: 0, kind: 1`. As the spec brief promises notes/memories/highlights/clips signing, this row grows. The list is dynamic, derived from the wiring contract in §9.
- **Relays panel.** Mirrors `FeedbackRelayClient.profileRelayURLs` plus any user-added entries. Status dot left of URL: filled `success` green (acked within 5min), filled `warning` amber (>5min, <24h), hollow grey (no record / never seen). RTT is the last successful publish round-trip in ms; dash if unknown. `↻` is a per-relay manual republish (re-signs the current kind-0 and pushes only to that one). This is intentionally noisier than the consumer brief — the user *wants* to see which relay didn't ack.
- **Sign out** is a red bordered button, full-width, at the bottom — never floating. Confirm dialog uses `.alert` (not `.confirmationDialog`) per the existing `UserIdentityView` comment about iOS 26 popover-promotion eliding Cancel.
- Tier: page is T0 reading; section dividers are `hairline`; the relays panel is the only place glass might creep in via row-pressable T1 hovers — keep it T0 for legibility.

## 5. NIP-46 Connect — Power Extensions

The existing `Nip46ConnectCard.swift` is a single-state card with one bunker URI input and one status string. Power users need a relay-by-relay table, manual reconnect, and per-request visibility. **Extend in place** — same file — with a disclosed `▾ Advanced` section.

```
┌──────────────────────────────────────────────────────────────┐
│  NIP-46 REMOTE SIGNER                            [Connected]│
│  ──────────────────────────────────────────────────────────  │
│  bunker     71a3…f29c   (Amber)                              │
│  user pub   3f2c1a8e…b41a9d0e                                │
│  permissions  sign_event:0, sign_event:1, get_public_key      │
│                                                              │
│  RELAYS  ────────────────────────────────────────────────────│
│  URL                              RTT   last sign  pending  ↻│
│  wss://relay.nsec.app             42ms  38s ago    0       ↻│
│  wss://relay.damus.io            128ms  2m ago     0       ↻│
│  wss://nostr.wine                  ✗    never      —       ↻│
│                                                              │
│  PENDING                                                     │
│  ◯  no requests in flight                                    │
│                                                              │
│  ▾ Advanced                                                  │
│  session pubkey   c8a1…0b3d  (ephemeral, regenerated on…)    │
│  bunker URI       bunker://71a3…f29c?relay=wss://relay.…    │
│                   [Copy]                                     │
│  auth challenges  last: 2026-04-30 14:02  (approved)         │
│                                                              │
│  [ Reconnect all ]   [ Disconnect bunker ]                  │
└──────────────────────────────────────────────────────────────┘
```

**State extensions** — these augment `RemoteSignerState` (no new variants required for MVP; the panel reads from existing transport):

- `RELAYS` table: per-relay status. Source the RTT and "last sign" by instrumenting `RemoteSignerTransport.swift` with an `@Observable` per-relay record (URL, lastSendAt, lastAckAt, lastRTT, pendingCount). Power users want this; consumer users will never look at it.
- `PENDING` row: when `signer.sign(...)` is in flight, show `kind: 1 · 1.2s · waiting on bunker ack`. If it resolves to `awaitingAuthorization(URL)`, the row swaps to the existing `authChallengeRow` UI **inline** (no full-card state replacement) — composition, not a switch statement.
- `↻` per-relay reconnect: drop and re-establish the WebSocket for that single relay only. The bunker connection persists across the others.
- `▾ Advanced` reveals: session keypair pubkey, full bunker URI (with `[Copy]` to allow re-pasting into another device), and the auth challenge log (last 5 entries).
- `[ Reconnect all ]` is a global reconnect — destructive against the WebSocket pool but non-destructive against the keychain `RemoteMeta`.
- **QR scan for bunker URI**: in the disconnected state (existing `inputRow`), add a third action below the text field next to `Connect bunker`: `[ Scan QR ]` opens the system camera, parses any string starting `bunker://`. Same handler — `connectRemoteSigner(uri:)`.

**Tier**: card stays at T1 (chrome of the Identity surface), the auth-challenge inline row borrows `glass.agent` tint at T2 — the same defended exception as the Settings entry badge.

## 7. First-Launch Experience

No welcome flow. The auto-generation already happens silently in `UserIdentityStore.start()` (`generateGeneratedKey()`). Surface it once and exit.

```
                                  ┌────────────────────────────┐
                                  │ ◐  Identity created         │
                                  │   bright-signal-a3f2        │
                                  │   npub1pl7…q9k4 · Local    │
                                  │              [ Customize ] │
                                  └────────────────────────────┘
```

- Top-anchored toast (T2 contextual tint per ux-15 §6), 6s auto-dismiss, swipe-up to clear early.
- `[ Customize ]` deep-links to the Identity Root view (§3).
- Fires once, gated on a UserDefaults flag `identity.firstLaunchToastShown`. If the user signs out and starts over, the flag is preserved — repeat visits get no toast. Power users who sign out know what they did.
- Haptic: `Haptics.light` (one tap). No sound. This isn't a celebration; it's a receipt.
- Does **not** appear when the user imports an nsec or connects a bunker — those are explicit acts and don't need acknowledgment.

### 8.3 Never signs

| Site | File:Line | Why |
|---|---|---|
| Agent-authored notes/messages | `App/Sources/Agent/LiveTENEXDelegationBridge.swift:90` | Uses an *agent* keypair. The task brief explicitly says "agent-authored content does not" sign with the user identity. |

### 8.4 Migration note (multi-account, v2)

Today the keychain layout is singular: `service: io.f7z.podcast.user-identity, account: user-private-key-hex`. For v2 multi-account, the account string becomes `user-private-key-hex.{fingerprint}` and a new `account: active-fingerprint` pointer holds which one is selected at boot. Existing single-account installs migrate by computing the fingerprint of the existing key on first launch under v2, renaming the keychain entry, and setting the pointer. Document in the Engineer's brief; out of scope for this Designer brief.

## 9. Glass Tier Discipline (summary)

Per ux-15. This brief touches T0, T1, and one defended T2:

- **T0 reading** — Identity root, profile editor (field rows), key management, audit log rows.
- **T1 chrome** — toolbars, filter chip rails, NIP-46 card body, confirm sheets.
- **T2 `glass.agent`** — exactly two surfaces: (a) the `Bunker via X` mode badge, (b) the inline `awaitingAuthorization` row in the NIP-46 card. Defended in §2: remote signing is *agent-mediated infrastructure*. Nothing else gets this tint.
- **Never** `accent.player` (copper) — identity is not now-playing.
- **Never** `accent.friend` (amber) — the user is not their own Nostr friend; amber is for incoming friend provenance only.

## 10. Open Questions

1. **Fingerprint hash** — proposed `sha256(hex)` truncated to first 16 chars. Alternatives: `sha256(npub)`, raw `pubkey[:8]`, or `crc32`. Pick one and lock it; surface across audit log, key management, multi-account chooser, and friend mention chips.
2. **Generated kind-0 at remote-signer adoption** — today auto-publish only fires for generated *local* keys. Should connecting a bunker that lacks a kind-0 trigger a "publish a default kind-0 via bunker?" prompt, or is that aggressively presumptuous? Recommend prompt, not auto.
3. **Audit log retention** — proposed FIFO at N=10,000. Tunable in advanced settings, default capped to keep keychain/UserDefaults footprint bounded.
4. **NIP-65 outbox model** — none of the relay logic in this brief uses kind:10002 relay lists. Power users expect it. Future work: read user's kind:10002 on adopt, override hard-coded `FeedbackRelayClient.profileRelayURLs`. Out of scope here, flagged for the engineer brief.
