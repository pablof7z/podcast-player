# iOS Build Pipeline

> The iOS build pipeline requires manual Rust lib rebuilds before Xcode builds, Tuist project generation for new files, and awareness of cache/clean semantics.

## Build Prerequisite

Xcode's incremental build system does not detect changes in external Rust static libraries. The Rust library must be linked as a static archive (.a) rather than a cdylib (.dylib) to prevent missing-library crashes on the device caused by embedded Mac-absolute paths. When building for an iOS device, any Rust cdylib (.dylib) files must be deleted so the linker falls back to the static library (.a), preventing embedded Mac-absolute paths that cause launch crashes on the device. Deploying to an iPhone requires performing the following steps in order: build the Rust library, delete the generated dylib, build the Xcode project, install, and launch. After modifying Rust source in the podcast crate, you must manually rebuild the iOS simulator static library before building in Xcode:
```bash
cargo build -p nmp-app-podcast --target aarch64-apple-ios-sim
```
Failing to do this results in the app linking a stale binary — the Rust changes won't take effect.

The Rust `build.rs` must correctly map the `aarch64-apple-ios-sim` target triple to the simulator SDK and use `xcrun` for SDK resolution rather than hardcoded paths. This applies to all compilation steps in `build.rs`, including the chkstk stub compilation, which must use `xcrun` and the correct target triple mapping rather than a hardcoded Xcode path and partial string match.

## Library Search Path

Xcode searches for the static library in this order:
1. `$(SRCROOT)/target/aarch64-apple-ios-sim/debug` (project-local target dir)
2. `~/.cargo/target-shared/aarch64-apple-ios-sim/debug` (shared `CARGO_TARGET_DIR`)

The shared target dir (`~/.cargo/target-shared`) is the canonical location. The project-local path is a fallback. <!-- [^14943-33] -->

## Incremental Build Cache

Xcode may use a cached build and not relink against a newly-rebuilt Rust library even when the binary timestamp changes. When in doubt, clean the Xcode build products before rebuilding:
```bash
# Via Xcode MCP: mcp__xcode__clean
# Or via xcodebuild: xcodebuild clean -workspace Podcastr.xcworkspace -scheme Podcastr
```
Then run `build_sim` with `preferXcodebuild: true` for a full rebuild. <!-- [^14943-34] -->

## Rebuilding After Cache Clear

If the shared `~/.cargo/target-shared` directory is cleared (e.g., to reclaim disk space), the iOS simulator static library must be rebuilt from scratch:
```bash
cargo build -p nmp-app-podcast --target aarch64-apple-ios-sim
```
This can take several minutes as it compiles the full NMP dependency graph for the simulator target. <!-- [^14943-35] -->

## Tuist Integration

The Xcode project is generated by Tuist (`tuist generate --no-open`). When new Swift files are added, Tuist must be re-run to include them in the Xcode project; otherwise the build will fail with missing file references. The workspace file is `Podcastr.xcworkspace`, the scheme is `Podcastr`, and the bundle ID is `io.f7z.podcast`. <!-- [^14943-36] -->

## Worktree Builds

Worktrees build into their own DerivedData folder with a different hash (e.g., `Podcastr-geippxazsjqayvewadgcfamxyzxd`). When installing a build from a worktree, use the worktree's DerivedData path, not the main repo's — the main repo's build is stale relative to worktree changes. <!-- [^14943-37] -->
