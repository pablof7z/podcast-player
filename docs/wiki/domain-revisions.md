# Domain Revisions

## Per-Domain DomainRevs

Per-domain typed sidecars replace the dead generic podcast.snapshot projection, delivering ~1KB deltas instead of multi-MB full-library pulls, eliminating the 1 Hz whole-library rebuild during playback. Per-domain sub-projections use register_typed_snapshot_projection with Option-gating so unchanged domains emit None and are omitted from the frame entirely, realizing the 10x performance fix. The per-domain domain set is: podcast.library, podcast.playback, podcast.downloads, podcast.inbox, podcast.settings, podcast.identity, podcast.widget, podcast.social, and podcast.misc (catch-all). DomainRevs provides per-domain rev counters scoped to each substate via Infra::with_domain(Domain), so a substate's existing bare self.infra.bump() routes correctly to both the domain counter and global rev with zero call-site churn. Domain sub-projections require real bump_domain calls at mutation sites, not just in tests, to actually fire per-domain deltas. PodcastHandle.handle.rev is Arc<AtomicU64> which implements ChangeGate, enabling register_snapshot_projection_gated('podcast.snapshot', Arc::clone(&handle.rev) as Arc<dyn ChangeGate>, ...) to skip re-serialization when rev is unchanged. The register_snapshot_projection_gated('podcast.snapshot') block was deleted because the v0.5.0 typed-first encoder discards the generic projections map, making it pure actor-thread waste. Slice-local domain payload builders only rebuild the domains whose rev advanced, extracting per-episode EpisodeSummary construction into a shared episode_summary helper used by both the full-library path and the queue builder, ensuring byte-identity by construction rather than field-by-field copying. Each domain envelope carries the global update.rev (not the per-domain counter) as its rev field; the per-domain counter is only the Rust-side emit gate, so Swift/Android trackers and lastProcessedRev live in the same rev namespace. Domain sub-projections use tombstone-on-empty semantics: {"rev":N,"<field>":null} clears the corresponding slice and advances the tracker, so shells can learn "downloads cleared / signed out / unsubscribed-all" and empty domains don't rebuild every tick. Direct @MainActor tombstone tests exercise mergeDomainFramesImpl and verify library/downloads/widget tombstones clear their slices. The signed_events typed FlatBuffer sidecar is decoded and injected under v.projections["signed_events"] in nmp_app_podcast_decode_update_frame, fixing the post-v0.3.0 regression where every nmp_app_sign_event_for_return result was silently dropped. The iOS push path was dead since NMP v0.3.0: KernelBridge.decode(pointer:) required v.projections["podcast.snapshot"] which was never encoded, causing every push frame to fail decode and fall back to 100% pull-driven operation. The Android push path was actively harmful: SnapshotCodec.decodeEnvelope decoded the slim v directly as PodcastSnapshot with all fields defaulted, causing every kernel emit to clobber the UI to an empty state. The empty-clobber bug on Android is fully removed: a frame with no domains returns null and never touches state, and blanking now requires an explicit library tombstone at a higher rev. iOS per-domain push consumption decodes PodcastDomainFrames from the projection sidecar, maintains a compositeUpdate merged from selective domain updates, and uses a DomainRevTracker for per-domain rev-monotonic drop guards, falling back to pull on cold start. The iOS push path keeps pull as the cold-start/fallback, so a broken domain envelope degrades to the old pull behavior rather than blanking the UI. The SettingsSnapshot decode uses a hand-rolled init(from:) with decodeIfPresent for every field, so it physically cannot throw keyNotFound — it is the identical decode path the pull already used, not a regression. A cold-start pull re-seed insurance uses a hasHydrated flag: on the first pull, the rev guard uses >= instead of >, and after that pull lands, the normal > monotonic guard is restored for all subsequent push and pull frames. Android domain-frame consumption uses @SerialName for all snake_case fields, a per-domain DomainRevTracker with remember { DomainRevTracker() } that resets on process restart and a drop guard with per-domain rev > tracker monotonicity, and absent domains are never cleared on merge. The 304 Not Modified response no longer bumps the snapshot rev, eliminating the full re-serialize + FFI decode that previously occurred on every unchanged feed during refresh-all. Silent half-applied writes on poisoned lock must return {ok:false, error:'store poisoned'} instead of {ok:true}, and rev must not be bumped on a swallowed write. Each domain builder currently calls full build_podcast_update per changed domain per tick (up to 7 full builds/tick), which is acceptable post-mitigation (~7.6ms each) but flagged as a kernel follow-up to share one build per tick across domain closures. Owned-podcast kind:54 episode backfill runs in the kernel via self-enqueue (per-episode publish_episode actions), not a synchronous loop, to avoid blocking the actor on N Blossom uploads. NIP-10 Nostr conversations are a kernel-owned projection on the podcast.social domain (the 8th registered domain), grouping inbound AgentNoteSummary records and OutboundTurn entries by root_event_id, merging turns in timestamp order and sorting conversations newest-first, with trusted computed live at projection time against ActiveFollowSet. The podcast.social domain sidecar must bump domain_revs.social via the real Infra::bump() path at action handler sites, never via manual fetch_add in tests — tests must drive the action through the real handler and assert the sidecar re-emits. The podcast.social domain uses skip_serializing_if = "Vec::is_empty" on the nostr_conversations wire field so the golden fixture remains byte-identical. The conversation projection's nostrConversationFromDTO maps wire rootEventId (snake_case) to domain rootEventID (uppercase D), Int timestamps to Date, and string direction to Direction enum in Swift. Android renders no social/conversations/friends/follow UI despite the kernel emitting podcast.social and DomainFrames.kt decoding it; this is the clearest feature-parity gap. The AgentSnapshot Kotlin mirror was corrected from the stale ConversationsSnapshot shape (active_count/pending_approvals) to the real shape (messages + is_busy + AgentMessageSummary). The real-path re-emit tests for social actions dispatch ApprovePeer/BlockPeer through the actual PodcastHostOpHandler::handle_social_action and assert the podcast.social sidecar re-emits with flipped trust, never a manual domain_revs.social.fetch_add. The podcast.misc domain split is explicitly deprioritized because the expensive build_misc_payload only rebuilds on low-frequency mutators (wiki/knowledge/picks), while high-frequency mutators (agent-chat streaming, voice, clips) bump only the global rev and never advance misc — so #425 already eliminated the hot path.
