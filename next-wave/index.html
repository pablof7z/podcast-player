Execution Principles

  1. Do not mark a scenario executed from this manifest alone; attach simulator/device evidence first.
  2. Every screenshot set must include source commit, app build, device/OS, appearance, Dynamic Type, Reduce Motion, Reduce Transparency, network mode, and redaction state.
  3. Glass and material critique must focus on navigation or control chrome only; dense content surfaces need readable solid or semantic backgrounds.
  4. Every actionable defect found during execution must be filed before a scenario can leave incomplete.

Common Capture Requirements

  1. Capture at least one step screenshot, one semantic UI tree, one command/log reference, and one generated report-page viewport check for every scenario.
  2. Run light appearance first; add dark appearance when the flow includes persistent navigation chrome, sheet chrome, or visual state that could invert poorly.
  3. Verify 44 pt minimum tap targets, safe-area reachability, Dynamic Type behavior, VoiceOver labels, contrast, Reduce Motion, and Reduce Transparency before scoring UI/UX above 2.
  4. Record NMP/RMP boundary observations: Rust-owned state/projections, thin native rendering, replayable clocks, and fail-closed provider or network behavior.
